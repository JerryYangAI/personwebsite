The eternal debate
📌 本节核心要点

MCP 协议 2024 年由 Anthropic 发布后，开发者社区一直在争论一个问题——"我该给我的 coding agent 配 MCP 工具，还是让它直接调 CLI？" Twitter 上经常吵——"MCP 是糟糕的工具调用接口" vs "MCP 是开放标准，未来必然胜"。这些争论大多数没有数据支撑。Mario Zechner 决定用自己的工具 terminalcp——它同时提供 MCP server 和 CLI 接口，可以同任务、同模型、不同工具地做 A/B 测试。他设计 120 次评测，得出 4 个洞察，最后给出一个让我反复回味的结论——"the protocol matters less than the tools you choose to build"。

CHAPTER 02 · TERMINALCP

terminalcp · 一个工具的两种形态

terminalcp two interfaces
📌 本节核心要点

terminalcp 解决的问题——普通 bash 在每次工具调用之间不能保持状态——你无法运行 "npm run dev" 让它持续，并跟它互动。Mario 写 terminalcp——持久 shell 会话管理器。提供 11 个核心操作——start、stop、list、stdin、stdout、stream、kill、resize、version、start_telnet、stop_telnet。同样的能力——既可用 MCP 工具调用，也可作为 CLI 命令。tmux/screen 也提供类似能力但是给人类用——terminalcp 是专为 LLM 设计的——无需 escape sequence、无前景终端假设。

terminalcp 解决的问题——普通 bash 工具在每次工具调用之间不保持状态。你无法运行 npm run dev 让它持续运行，然后跟它的输出互动。这对 web 开发是致命的限制。

Mario 的解决方案——持久 shell 会话管理器。11 个核心操作——

🛠 terminalcp 11 个操作
  • start——启动一个新进程并返回 process ID
  • stop——停止指定进程
  • list——列出所有当前活跃会话
  • stdin——向进程发送输入
  • stdout——读取进程的输出
  • stream——流式监控输出
  • kill——强制杀掉进程
  • resize——调整终端尺寸
  • version——获取版本
  • start_telnet、stop_telnet——telnet 接口（人类调试用）

关键 trick——同样的能力，提供两种暴露方式——

· MCP server——通过 MCP 协议暴露这 11 个工具调用
· CLI——通过命令行调用——例如 terminalcp start "npm run dev"、terminalcp stdout <pid>

这让同任务下 MCP vs CLI 的对比变得严格——底层操作完全一样，只是 agent 通过什么接口调它。

tmux 和 screen 提供类似能力——但它们是给人类用的——需要复杂的escape sequence，假设有前景终端。terminalcp 是专为 LLM 设计的——纯文本接口、简洁参数、无终端假设。

CHAPTER 03 · EVALUATION DESIGN

评测设计 · 3 任务 × 4 工具 × 10 重复

Evaluation design
📌 本节核心要点

3 个真实任务——(A) Web Dev 任务——启动 npm run dev，与浏览器交互，处理日志；(B) Python 数据科学交互式 REPL——分析超过 8 万样本的数据集，处理 Pandas 的 deprecation warnings；(C) 项目分析任务——克隆 libGDX 仓库（10 万+ 行 Java），分析"哪些子项目用了 LWJGL3"。4 种工具方式——(1) 直接 bash（每次新启 shell）；(2) tmux；(3) screen；(4) MCP terminalcp。每组 10 次重复以求稳定。整个评测花了 8 小时、约 50 美元。

Mario 选了 3 个真实编码任务——每个都反映了 agent 工程师常见的实战场景——

📊 三个任务
  • (A) Web Dev 任务——启动 npm run dev、与浏览器交互、处理日志输出。典型的"长寿命进程 + 持续交互"场景。
  • (B) Python 数据科学——在 Python REPL 里分析 80K+ 样本数据集，处理 Pandas deprecation warnings。典型的"交互式探索 + 状态保持"场景。
  • (C) 项目分析——克隆 libGDX 仓库（10万+ 行 Java 代码），分析"哪些子项目用了 LWJGL3 而非更老的版本"。典型的"大代码库探索"场景。

4 种工具方式——

· 直接 bash——Claude Code 的标准 Bash 工具，每次调用新启 shell
· tmux——通过 tmux 维持持久 session
· screen——通过 screen 维持持久 session（老牌 Unix 工具）
· MCP terminalcp——通过 MCP 协议调 terminalcp 的工具

每组 10 次重复——以求稳定的成功率和 token 用量数据。

整个评测花了 8 小时、消耗 约 50 美元 Claude API tokens。Mario 把全部 raw logs 都开源在 terminalcp GitHub 仓库。这种程度的实证主义在 AI 社区非常稀缺。

CHAPTER 04 · RESULTS

完整结果表 · 成功率、时间、tokens

Full results
📌 本节核心要点

三个任务的结果——(A) Web Dev——四种方法成功率都 100%，MCP 最快（44.6 秒）、最便宜（$0.20）；(B) Python 数据科学——四种方法成功率都 100%，MCP 略快（59.4s）但 tmux 更便宜（$0.17）；(C) 项目分析——这是真正的分水岭——bash 100%、tmux 80%、MCP 70%、screen 0%！Mario 发现——screen 的设计问题让它在大代码库探索任务上完全失败。综合——MCP 比 bash 整体快 23%、便宜 2.5%，但不是革命性差距。

三个关键观察——

(1)——对于 Web Dev 和 Python 任务，所有 4 种方法都 100% 成功。MCP 略快略便宜，但不到一倍的差距。这驳斥了 "MCP 显著更好" 或 "CLI 显著更好" 两种极端论调。

(2)——screen 在项目分析任务上 0% 成功率！ Mario 调查原因——screen 在大量输出时会 truncate buffer，导致 Claude Code 看不到关键的 grep 结果或目录结构。这是 screen 的设计缺陷暴露在 LLM 工作流下——它是为人类设计的，人类可以 scroll back，LLM 不能。

(3)——项目分析任务上 bash 反而是冠军（100%）——因为 bash 的"每次新启 shell" 模式不会有 session state 污染——简单的工具在简单任务上反而最 robust。

CHAPTER 05 · THE SECURITY FINDING

最惊人的发现 · MCP 绕过安全检测

The security finding
📌 本节核心要点

这是这篇文章最重要、最不应被忽视的部分。Mario 跑评测时偶然发现——"MCP 路径下 Haiku tokens 是 35K，CLI/bash 路径下是 1.3-2M tokens"——5 个数量级的差距！ 调查后才知道——Claude Code 在每次 Bash 工具调用前会用 Haiku 跑一遍"恶意命令检测"，但MCP 工具调用完全绕过这个检测！这意味着——"装上 MCP server 等于在 Claude Code 的安全防线上开了一个洞"。Mario 写——"I'm not sure this is the intended behavior"。这是整个 MCP 生态需要直面的安全设计问题。

Haiku Token Anomaly Malicious Cmd Detection Security Gap

这是这篇文章最重要、最不应被忽视的部分。我反复读了 3 遍才确信 Mario 不是在开玩笑。

他跑评测时偶然发现——

看完整 tokens 用量时——
我注意到一个奇怪的差距——

而 CLI/bash 路径下是 1.3-2 million tokens。

— THE 5-ORDER-OF-MAGNITUDE GAP

Mario 调查原因后才发现——Claude Code 在每次 Bash 工具调用前，会用 Haiku 模型跑一遍"恶意命令检测"——检查 agent 是不是要做危险事（rm -rf /、外发敏感数据、修改系统文件⋯⋯）。

但——MCP 工具调用完全绕过这个检测！

"I'm not sure this is the intended behavior"——Mario 在文章里这样写。

这意味着——"装上 MCP server 等于在 Claude Code 的安全防线上开了一个洞"。任何 MCP 服务器只要它最终调到 shell（terminalcp、shell-mcp 之类）——就可以绕过 Anthropic 精心设计的 Haiku-based 安全网。

这不是 Mario 在"找茬"——他是诚恳地把数据展示出来，让 Anthropic 和社区都注意到这个差距。这种诚实的工程师精神是这篇文章值得记的原因之一。

未来——MCP 网关层（比如上一篇 Cloudflare MCP Portal）很可能要把这一层安全检测重新加回来——这是 2026 年企业 MCP 部署的必修课。

CHAPTER 06 · KEY FINDINGS

4 个关键洞察

Four key findings
📌 本节核心要点

Mario 从 120 次评测里浓缩出 4 个洞察——(1) MCP 和 CLI 在大多数任务上等价——成功率都 100%，速度/成本差距 < 30%；(2) 工具设计比协议选择更重要——screen 因为人类导向的 buffer 设计在 LLM 工作流崩了；(3) Claude Code 的安全检测有重大漏洞——MCP 完全绕过；(4) 实测必不可少——所有这些发现都不可能从纯理论或者 Twitter 争论里得出。Mario 给同行的实用建议——"如果你已经在用 MCP，继续用；如果你在用 CLI，也继续用"——但要确保测试自己的真实任务。

Mario 在文章末尾把 120 次评测的洞察浓缩成 4 条——

🎯 四个关键洞察
  • (1) MCP 和 CLI 在大多数任务上等价——成功率都 100%，速度/成本差距 < 30%。"The protocol choice doesn't dramatically affect outcomes"。
  • (2) 工具设计比协议选择更重要——screen 因为人类导向的 buffer 设计在 LLM 工作流崩了——不是 protocol 问题，是工具是否为 agent 设计的问题。
  • (3) Claude Code 的安全检测有重大漏洞——MCP 完全绕过——"This deserves attention from the Claude Code team"。
  • (4) 实测必不可少——所有这些发现都不可能从纯理论或 Twitter 争论里得出——"build it, measure it, share the data"。

Mario 给同行的实用建议——

"如果你已经在用 MCP，继续用——它工作。"
"如果你在用 CLI，也继续用——它也工作。"
"但要确保你测试自己的真实任务——你的 mileage may vary。"
"不要盲信我的数据——3 个任务太少。用你自己的代码库跑你自己的 agent。"

这种"我给你数据，你自己判断"的态度——跟 Anthropic 内部工程师 Thariq 那篇 "see like an agent" 是同一种哲学。

CHAPTER 07 · TOOL DESIGN OVER PROTOCOL

Mario 的结论 · 工具设计优于协议

Tool design over protocol
📌 本节核心要点

Mario 最终的结论——"the protocol matters less than the tools you choose to build"。这句话需要在 2026 年的 MCP 热潮里反复回响。MCP 协议本身没错——但"接到协议下面的工具"才是决定 agent 实际效果的关键。"如果你的工具是为人类设计的（像 screen），不管你包了多漂亮的 MCP server 还是 CLI 还是 GUI，它在 LLM 工作流下都不会好用"。这跟 Thariq 在 Anthropic 内部 X 帖子里强调的 "See like an agent" 是同一个工程哲学的两个面——设计工具时要把 agent 当成"用户"，而不是事后包装一个适配层。

Mario 最终的结论——

The protocol matters less
than the tools you choose to build.

协议本身没错——
但接到协议下面的工具，
才是决定 agent 实际效果的关键。

— THE CORE INSIGHT

这句话需要在 2026 年的 MCP 热潮里反复回响。

很多人讨论 MCP 时——把它当成 "AI 的 HTTP"，似乎只要"接上 MCP" 就万事大吉。Mario 用数据说明——"接上 MCP" 只是 trivial 的部分。真正决定 agent 表现的是——你提供的工具本身是不是为 agent 设计的。

screen 是反面教材——它是 1980 年代为人类设计的工具，包成 MCP 也好、CLI 也好、GUI 也好——buffer truncation 这种"对人类无伤大雅但对 LLM 致命"的设计问题永远存在。

terminalcp 是正面教材——它是 2025 年专门为 LLM 设计的工具——纯文本、简洁参数、没有终端假设、stdout 完整保留。包成 MCP 还是 CLI 都好用——因为底层工具本身是对的。

这跟 Thariq 在 Anthropic 内部 X 帖子里强调的 "See like an agent" 是同一个工程哲学的两个面——

· Thariq——设计工具时要把 agent 当成"用户"
· Mario——底层工具的设计决定 agent 表现，不要事后包装一个适配层就以为完事

这两个视角合起来，就是 2026 年 agent 工具设计的金科玉律。

CHAPTER 08 · EDITOR'S NOTE

广哥的总结 · 第一手数据值得反复读

First-hand data matters

读完这篇文章，我有几点感受——

第一——这是我 2025 年读过的最有价值的 AI agent 实证研究之一。整个 AI 社区充斥着"我感觉 MCP 更好" / "我感觉 CLI 更好"这种零数据论断。Mario 8 小时、50 美元、120 次重复跑出来的数据，就是这个领域稀缺的"第一手实证"。

第二——"MCP 绕过 Claude Code 安全检测"这个发现必须被 Anthropic 团队、被整个 MCP 生态认真对待。35K vs 1.3-2M Haiku tokens 的差距不会撒谎。这暗示——当前 MCP 协议设计在"安全策略统一应用"这一层有明显空白。Cloudflare MCP Portal 把 Gateway 加进来已经是朝正确方向的一步——但Anthropic 自己也应该考虑在 MCP 工具调用前加同样的 Haiku 检测。

第三——"工具设计比协议选择更重要"这句话是给所有 AI agent 工程师的。你花再多时间纠结"用 OpenAI Functions、用 MCP、还是用 Tool Use API"——都不如花时间想清楚 "我提供给 agent 的工具是不是为 agent 设计的"。如果你的工具是"把现有 CLI 套一层 MCP wrapper"——那你大概率重蹈 screen 的覆辙。

第四——"成本差距 < 30%、成功率几乎都 100%"这个数据让我想起来——2025-2026 年的 agent 工具选择已经不再是"对错"的问题，而是"细微优化"的问题。这是领域成熟的标志——就像 2010 年代"NoSQL vs SQL" 最后变成了"each has its place"一样。

第五——把这篇文章和 Thariq 的 X 帖子、Cloudflare MCP Portal 文档放在一起读——你能看到 2025-2026 年整个 AI agent 工程界正在从"实验性混乱"走向"工程纪律"。Anthropic 工程师写工程笔记（Thariq）、大厂做基础设施层（Cloudflare）、独立工程师做实证基准（Mario）——每一种贡献都在让这个领域变得更可信。

也不是数据。

120 次重复跑出来的成功率、时间、tokens——
