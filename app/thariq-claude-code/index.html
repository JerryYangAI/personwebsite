CHAPTER 01 · INTRODUCTION

引言 · 数学题与计算机的类比

A math problem and a computer
📌 本节核心要点

Thariq 开篇问了一个核心问题——"如何设计你 agent 的工具？" 他用一个让人会心一笑的类比——给你一道难数学题，你需要什么工具？纸是最低标准（被手工计算限制）；计算器更好（但需要会操作高级功能）；电脑最快最强（但需要会写代码）。同样的逻辑——给 agent 的工具必须和它自己的能力相匹配。怎么知道这些能力？"You pay attention, read its outputs, experiment. You learn to see like an agent."

Thariq @trq212 Action Space Design See Like an Agent

2026 年 2 月 28 日，Thariq——推特 @trq212，Anthropic Claude Code 团队工程师（之前在 YC W20、South Park Commons、MIT Media Lab）——在 X 上发了一篇长帖。3 天390 万阅读，1.1 万转发，1.6 万点赞。

开篇一句话就把整个工程困境拎出来——

构建一个 agent harness 最难的部分之一，
就是设计它的"行动空间"（action space）。

Claude 通过 Tool Calling 行动，
但 Claude API 里能用很多原语构造工具——
bash、skills、最近的 code execution⋯⋯

给定这些选项——你怎么设计你 agent 的工具？
需要 1 个工具（比如 code execution 或 bash）？
还是 50 个工具（每个用例一个）？

— THARIQ, 2026/02

Thariq 给的思考框架很美——把自己放进模型的脑子里——

"想象你被给了一道难数学题。你需要什么工具来解？这取决于你自己的技能！"

纸——是最低标准，但被手工计算限制。
计算器——更好，但你需要知道怎么操作高级功能。
电脑——最快最强，但你需要会写代码。

"这是一个有用的框架来设计你的 agent。你想给它形状跟它自己能力相匹配的工具。但你怎么知道这些能力？You pay attention, read its outputs, experiment. You learn to see like an agent."

"像一个 agent 一样看"——这句话是整篇文章的灵魂。下面的所有故事都在演示这种"观察 → 调整 → 再观察"的循环。

CHAPTER 02 · ASKUSERQUESTION

AskUserQuestion · 3 次尝试的演化

3 attempts at elicitation
📌 本节核心要点

这是一个非常典型的"工程师笔记"故事——团队想改进 Claude 提问的能力（elicitation）。Claude 当然可以用纯文本问问题——但"回答这些问题感觉花了不必要的时间"。他们做了 3 次尝试——(1) 修改 ExitPlanTool 加 questions 数组——失败，模型混淆"plan vs questions"；(2) 改输出格式让 Claude 输出特定 markdown——失败，模型不保证遵守格式；(3) 新建 AskUserQuestion 工具——成功。关键是"Claude 似乎喜欢调这个工具"——"即使最好设计的工具，如果 Claude 不理解怎么调，也不工作"。

AskUserQuestion Elicitation ExitPlanTool

第一个故事——构建 AskUserQuestion 工具——目标是改进 Claude 提问的能力（elicitation，引发用户表达）。

背景——Claude 当然可以用纯文本问问题——但他们发现"回答这些问题感觉花了不必要的时间"。怎么降低这种摩擦，增加用户和 Claude 之间的沟通带宽？

3 次尝试——

🔄 三次迭代
  • 尝试 #1 · 修改 ExitPlanTool——加一个 questions 数组参数，让它在 plan 旁边问问题。最简单的实现。但失败——同时要 plan 和关于 plan 的 questions 会让 Claude 混淆。"如果用户的答案跟 plan 冲突怎么办？Claude 需要调 ExitPlanTool 两次吗？" 需要新方法。
  • 尝试 #2 · 改输出格式——修改 Claude 的输出指令，让它服务一种稍微修改的 markdown 格式——比如要求"项目符号问题，可选项放在括号里"。然后解析这个格式渲染 UI。"最通用的改动"，Claude 似乎做得还行。但不保证——Claude 会加多余句子、省略选项、或用完全不同的格式。
  • 尝试 #3 · AskUserQuestion 工具——最后他们做了一个 Claude 可以在任何时候调用的工具——特别在 plan mode 被 prompt 去用。工具触发时显示 modal 显示问题，阻塞 agent 的循环直到用户回答。

这个工具让他们能prompt Claude 产出结构化输出，确保 Claude 给用户多个选项。它也让用户能组合这个功能——比如在 Agent SDK 里调或在 skills 里引用。

但 Thariq 最强调的不是技术细节——

最重要的是——
Claude 似乎喜欢调这个工具，
我们发现它的输出工作得很好。

即使最好设计的工具，
如果 Claude 不理解怎么调，也不工作。

— THE FUNDAMENTAL INSIGHT

这是工程师才看得见的真相——"模型喜欢用"是工具设计的必要条件。你设计得再优雅，如果模型不调，就是死代码。这个标准很难量化，但 Anthropic 工程师每天都在用直觉去判断。

Thariq 末尾留下一个开放问题——"这是 Claude Code 里 elicitation 的最终形态吗？我们不确定"。下一节的故事正好说明——对一个模型工作的，不一定对另一个最好。

CHAPTER 03 · TODOS TO TASKS

从 TodoWrite 到 Task Tool · 跟着模型能力升级

Todos to Tasks
📌 本节核心要点

Claude Code 首发时，他们意识到模型需要Todo 列表保持任务专注——所以给了 Claude 的 TodoWrite 工具。但模型还是常常忘记自己要做什么——所以每 5 轮插一个系统提醒。但当模型变强后——它们不只不需要被提醒，反而被限制了。Opus 4.5 还在子代理使用上变得更好——但子代理之间怎么协调一个共享 Todo 列表？他们用 Task Tool 替换了 TodoWrite——Tasks 关于"让 agent 之间通信"而非"让模型不跑偏"。"随着模型能力提升，模型曾经需要的工具可能现在限制它们。"

TodoWrite Task Tool Opus 4.5 Subagent Coordination

第二个故事——从 TodoWrite 演化到 Task Tool——展示了模型能力升级如何让旧工具变成限制。

Claude Code 首发时，团队意识到模型需要 Todo 列表保持专注。Todos 可在开始时写下，模型做工作时勾掉。TodoWrite 工具就是这样诞生的——能写或更新 Todos，显示给用户看。

但即使如此——他们经常看到 Claude 忘记自己要做什么。适应方法——每 5 轮插入系统提醒提醒 Claude 它的目标。

然后模型变强了——

但随着模型改进——
它们不只不需要被提醒 Todo 列表，
反而被它限制。

被发提醒让 Claude 觉得它必须坚持原列表——
而不是修改它。

我们还看到 Opus 4.5 在用子代理上变得好得多——
但子代理怎么协调一个共享的 Todo 列表？

— TOOLS THAT BECOME LIMITS

看到这个，他们用 Task Tool 替换 TodoWrite——

· Todos 关于让模型不跑偏（保持单一 agent 在轨）
· Tasks 关于帮助 agent 之间通信——可以包括依赖、跨子代理共享更新、模型能修改和删除

这是典型的工程哲学转变——从"给单 agent 加 guardrails"到"给多 agent 系统加协调原语"。

Thariq 给出的普遍教训——

随着模型能力提升——
模型曾经需要的工具
可能现在限制它们。

不断重新审视"哪些工具是必须的"这一假设——是重要的。

这也是为什么坚持一小套能力大致相似的模型很有用——
更容易做这种维护。

— THE EVOLUTIONARY PRINCIPLE

这是整篇文章最有思想性的洞察之一——"工具是模型能力的函数"。所有 agent 系统都需要跟着模型能力的演化重新评估自己的脚手架。"模型能力提升 → 工具被审视 → 旧工具被替换"是必然循环。

CHAPTER 04 · SEARCH

搜索接口 · 从 RAG 到 Grep 的觉醒

From RAG to Grep
📌 本节核心要点

Claude 最重要的工具集之一是"搜索工具"——用来构建它自己的上下文。Claude Code 首发时，他们用 RAG 向量数据库——快、强，但需要索引和设置，在不同环境里脆弱。更重要——Claude 被给这种上下文，而不是自己找上下文。他们的觉醒——"既然 Claude 能在 web 上搜索，为什么不能在你的代码库搜索？" 给 Claude 一个 Grep 工具——它就能搜文件、自己构建上下文。"Claude 越聪明，给它正确工具时它就越擅长构建自己的上下文。"

RAG Vector DB Grep Tool Self-context Building

第三个故事——搜索接口的演化——展示了一个反 RAG 共识的觉醒。

Claude Code 首发时，他们用 RAG 向量数据库为 Claude 找上下文。RAG 强大快速——但有问题——

· 需要索引和设置——增加部署摩擦
· 在不同环境里脆弱——不同代码库要重新调
· 更重要——Claude 被给这种上下文，而不是自己找

Thariq 给出觉醒的那一刻——

但如果 Claude 能在 web 上搜索——
为什么不能搜你的代码库？

给 Claude 一个 Grep 工具——
我们就能让它搜文件、自己构建上下文。

这是我们看到的一个模式——
Claude 越聪明，给它正确工具时它就越擅长构建自己的上下文。

— THE GREP EPIPHANY

这是2024-2026 年 AI Agent 设计的一个重要转向——从"喂上下文"到"让 agent 自己找"。RAG 是"喂"的代表；Grep + 文件系统访问是"找"的代表。Claude Code 选了后者。

这有多重好处——(1) 部署简单（不需要建向量库索引）；(2) 跨环境 robust（在任何代码库都能用）；(3) Claude 知道自己为什么需要这个上下文（因为它自己问了）；(4) 不浪费上下文窗口（按需取，不是预先 dump）。

这个洞察后来变成更深的概念——"Progressive Disclosure"——下一章讲。

CHAPTER 05 · PROGRESSIVE DISCLOSURE

Progressive Disclosure · Skills 的递归发现

Progressive disclosure
📌 本节核心要点

引入 Agent Skills 时他们形式化了"渐进披露"——"allows agents to incrementally discover relevant context through exploration"。Claude 读 skill 文件，那些文件能引用其他文件，模型可以递归地读。skills 的一个常见用法——给 Claude 加搜索能力，比如"如何用某 API、如何查询某数据库"的指令。"在一年中，Claude 从'基本不能构建自己的上下文'变到'跨多层文件做嵌套搜索找到精确所需上下文'。" Progressive Disclosure 现在是不加工具而加功能的标准技术。

Agent Skills Progressive Disclosure Recursive Discovery

当 Anthropic 引入 Agent Skills 时，他们形式化了这个想法——"Progressive Disclosure"（渐进披露）——

渐进披露——
允许 agent 通过探索增量地发现相关上下文。

Claude 能读 skill 文件——
那些文件能引用其他文件——
模型可以递归地读。

事实上——skills 的一个常见用法——
给 Claude 加更多搜索能力——
比如给它指令怎么用某 API 或查询某数据库。

— PROGRESSIVE DISCLOSURE FORMALIZED

Thariq 给出一个让我合上文章想了很久的总结——

"在一年中，Claude 从'基本不能构建自己的上下文'，变到跨多层文件做嵌套搜索找到它需要的精确上下文"。

Progressive Disclosure 现在是一个常用技术——加新功能而不加工具。这是非常优雅的设计——"不在工具池里多塞一把锤子，而是给 agent 一本能扩展的指南"。

对照MBZUAI 那篇 Dive into Claude Code（我之前发过的）——它在第 7 章压缩管道里就提到 "CLAUDE.md lazy loading" 和 "Deferred tool schemas"——都是 Progressive Disclosure 原则的具体体现。两篇文章从不同角度都指向同一个工程哲学。

CHAPTER 06 · GUIDE SUBAGENT

Claude Code Guide Subagent · 不加工具的扩展

The guide subagent
📌 本节核心要点

Claude Code 现有约 20 个工具——加新工具的门槛很高，因为这给模型多一个要思考的选项。"context rot"是真实问题。具体例子——Claude 不知道关于 Claude Code 自己的足够（"如果你问它怎么加 MCP 或某 slash 命令做什么，它答不上"）。可以把所有信息塞进 system prompt——但用户很少问这种问题，会加 context rot 干扰主要工作。他们用了 Progressive Disclosure——给 Claude 一个文档链接能加载查询。但 Claude 会把大量结果塞进 context 找答案——而其实你只需要答案。解法是 Claude Code Guide Subagent——专门 prompt 在"用户问你自己"时调用，subagent 有详细搜索文档的指令，返回精炼答案。

Claude Code 现在有约 20 个工具。他们不断问自己"我们是否都需要"——加新工具的门槛很高，因为给模型多一个要思考的选项。

具体案例——他们发现 Claude 不知道关于 Claude Code 自己的足够——"如果你问它怎么加 MCP 或某 slash 命令做什么，它答不上"。

本可以把所有这些信息放进 system prompt——但用户很少问，会加 context rot 干扰 Claude Code 的主要工作——写代码。

第一次尝试——Progressive Disclosure——给 Claude 一个文档链接能加载查询。这能工作，但他们发现——Claude 会把大量结果塞进 context 找答案，而其实只需要答案。

所以最后他们建了 Claude Code Guide subagent——

🎯 Guide Subagent 设计
  • 触发——Claude 被 prompt 在用户问"关于 Claude Code 本身"时调用
  • 能力——subagent 有详细的"如何好地搜索文档"和"返回什么"的指令
  • 结果——返回精炼答案，不污染主 context
  • 关键——不加新工具——只加一个 subagent——但拓宽了 Claude 的"action space"

"虽然不完美——Claude 还会在你问它怎么设置自己时被搞糊涂——但比以前好多了。我们能在不加工具的情况下，给 Claude 的行动空间加东西"。

这是 Anthropic 工程师的"工程师才看得见的真相"——"加 subagent" 比 "加工具" 更便宜，因为 subagent 不占主 context window，专门化的指令在 subagent 内部。这是非常聪明的设计选择。

CHAPTER 07 · ART NOT SCIENCE

艺术多于科学 · Thariq 的结论

An art, not a science
📌 本节核心要点

Thariq 直接告诉读者——"如果你期望一套严格规则告诉你怎么建工具，不好意思，这不是那种指南"。设计 agent 的工具是艺术多于科学——"取决于你用的模型、agent 的目标、它操作的环境"。结论只有 5 个字——"Experiment often, read your outputs, try new things. See like an agent."

Thariq 的最后一节标题就是"An Art, not a Science"——这是这篇 X 长帖的诚实归宿——

如果你期望一套关于如何建你工具的严格规则——
不幸地，这不是那种指南。

设计你模型的工具——
是艺术多于科学。

它严重取决于你用的模型、agent 的目标、它操作的环境。

Experiment often, read your outputs, try new things.
See like an agent.

— THARIQ'S CONCLUSION

这个结论挺像 Anthropic 的文化味儿——不教条、不假设有银弹、把工程师的判断当成第一手洞察。

"Read your outputs"——这一句尤其重要。AI 工程师很容易陷入"我设计了这个抽象，所以模型应该用得好"的错觉。真相只能从读输出里看到——Claude 是真的喜欢调这个工具吗？输出有多干净？token 浪费在哪里？

"See like an agent"——把这句当成一种认知训练。每次设计工具时——把自己代入模型，问"如果我是 Claude，这个工具描述会让我想用吗？""这个输出会让我下一步做对决定吗？"

CHAPTER 08 · EDITOR'S NOTE

广哥的总结 · 工程师才看得到的真相

An engineer's truth

读完这篇短文，我有几点感受——

第一——这篇文章和我之前发的 《Dive into Claude Code》（MBZUAI 那篇）形成完美互补。MBZUAI 是外部研究者读源码的架构视角——5 个价值观、13 个设计原则、5 层子系统。Thariq 这一篇是内部工程师的第一手视角——4 个具体决策的演化故事。两篇一起读，你才看到 Claude Code 的"为什么"和"怎么做"。

第二——"Claude 似乎喜欢调这个工具"这种说法看起来不科学，但是 Anthropic 工程师的核心审美判断。这种把模型当成"用户"的视角，是 AI agent 设计跟传统软件设计最大的不同。传统软件——你设计 API 给开发者用。AI agent——你设计工具给模型用。模型有偏好、有摩擦、有"我喜欢这个"——你必须尊重这一点。

第三——从 TodoWrite 到 Task Tool 的故事给我一个深刻提醒——"今天的最佳实践可能是明天的限制"。所有 AI 工程团队都要建立"重新审视假设"的节律。每次模型大版本升级，整个工具集都该重新评估。

第四——"Progressive Disclosure"是这篇文章给我留下的最实用的架构原则。下次我自己设计 agent 系统——第一选择是 Skills + Lazy Loading，不是所有功能塞 system prompt 或者加新工具。

第五——"See like an agent"这句话值得贴在工位上。它反映了 Anthropic 内部的工程文化——不依赖框架、不依赖理论，依赖持续观察、读输出、做实验。这跟 Dario 在《Adolescence of Technology》里说的"我们建 Claude 像在养一个东西而不是造"是同一种哲学。

这篇 X 帖子 3 天 390 万阅读不是偶然——它是工业界 AI Agent 工程师都需要的"诚实的工程笔记"。
