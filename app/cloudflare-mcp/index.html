CHAPTER 01 · INTRODUCTION

引言 · 为什么需要 MCP 门户

Why an MCP portal
📌 本节核心要点

MCP（Model Context Protocol）是 Anthropic 2024 年发明的开放协议，2025 年捐给 Linux Foundation。但企业里用 MCP 面临大量痛点——多个服务器要分别管理、OAuth 流程多次重复、工具命名冲突、上下文窗口被工具定义占满、缺少集中审计和 DLP。Cloudflare One 推出的 MCP 服务器门户把这些痛点系统性解决——"把多个 MCP 服务器集中到一个 HTTP 端点"。用户通过 Cloudflare Access 登录门户 URL，系统提示对每个需要 OAuth 的服务器进行身份验证。这是MCP 从"开发者工具"走向"企业基础设施"的关键一步。

MCP Portal Cloudflare One Zero Trust Single HTTP Endpoint

2026 年 6 月 25 日，Cloudflare 文档更新——发布 MCP 服务器门户（MCP Server Portal）功能。它是 Cloudflare One 在AI 控件下的新产品。

文档开篇的一句话定调了整个产品——

MCP 服务器门户——
将多个模型上下文协议（MCP）服务器
集中到一个 HTTP 端点上。

— CLOUDFLARE ONE DOCS, 2026/06

看起来简单的一句话——但背后是企业 MCP 实践里积累的所有痛点——

· 一个企业可能有几十个内部 MCP 服务器——HR、CRM、Wiki、数据库、监控、CI/CD⋯⋯每个都要单独配置、单独认证
· 每个 MCP 服务器可能要自己的 OAuth 流程——开发者要在 IDE 里走几遍授权
· 多个服务器可能工具名冲突（两个服务器都有 search 工具）
· 工具定义全部塞进 Claude 上下文——大企业里能爆几十 K token
· 缺少集中审计——谁调了什么工具、参数是什么、返回了什么
· 缺少 DLP（数据丢失防护）——agent 可能把敏感数据拿出去

Cloudflare 的 MCP Portal 把所有这些痛点系统性解决——用 Cloudflare 已有的 Access + Gateway 基础设施做载体。这是 MCP 从"开发者工具"走向"企业级基础设施"的关键一步。

CHAPTER 02 · FEATURES

六大主要特点

Six main features
📌 本节核心要点

文档列了门户的 6 大功能——简化多 MCP 服务器访问（同时支持未鉴权和 OAuth 服务器，用户走一次 Cloudflare Access 登录）；每门户可定制工具（管理员选择给用户暴露哪些工具和提示——"AI 模型接触到的外部上下文越少，其响应往往就越好"）；工具和提示别名（无需修改上游服务器即可改名/描述）；上下文优化（最小化或隐藏工具定义减少上下文使用）；非浏览器客户端支持（OAuth 2.0 标准流程 + 服务令牌）；Code 模式（所有上游工具合并为一个 code 工具，AI 编写 JavaScript 调用类型化方法）；可观测性（Cloudflare Access 记录每个请求，可路由到 Gateway 做 HTTP 日志和 DLP 扫描）。

文档给的 6 大特点列得很清楚——

🔑 六大主要功能
  • (1) 简化对多个 MCP 服务器的访问——同时支持未鉴权和 OAuth 保护的 MCP 服务器（例如通过 Access for SaaS 或第三方 OAuth）。用户通过 Cloudflare Access 登录门户 URL，系统会提示对每个需要 OAuth 的服务器分别进行身份验证。
  • (2) 每门户可定制工具——管理员可根据特定用例定制门户，选择哪些工具和提示模板对用户暴露。"AI 模型接触到的外部上下文越少，其响应往往就越好"——这正是 Anthropic 在 Claude Code 工程实践里反复强调的原则。
  • (3) 工具和提示别名——管理员可在门户或服务器级别重命名工具和提示，编辑描述，无需修改上游 MCP 服务器。别名帮助用户找到合适工具，也帮助 AI 代理选择正确工具。
  • (4) 上下文优化——门户支持查询参数选项，可通过最小化或隐藏工具定义来减少上下文窗口使用。详见第 7 章。
  • (5) 非浏览器客户端支持——MCP 客户端通过托管 OAuth 使用标准 OAuth 2.0 授权码流程。非浏览器客户端收到 401 包含指向 Access OAuth 发现端点的 WWW-Authenticate 标头响应，而非浏览器重定向。也可使用 Access 服务令牌做机器对机器访问。
  • (6) Code 模式——所有门户默认启用代码模式。它把所有上游工具合并为一个 code 工具。AI 代理编写 JavaScript 代码调用每个工具的类型化方法，代码在隔离的动态 Workspace 环境中运行。无论有多少工具可用，上下文窗口使用情况都能保持固定。

再加上一个——可观测性——用户的 AI 代理连接到门户后，Cloudflare Access 记录使用门户工具发出的各个请求。可选择将门户流量路由到 Cloudflare Gateway，实现更丰富的 HTTP 日志记录和 DLP（数据丢失防护）扫描。

这一项对企业最关键——没有审计就没有合规、没有 DLP 就没有安全。Cloudflare 用自家已有的 Access + Gateway 把这个能力"白嫖"接进来。

CHAPTER 03 · FLOW

工作流程 · 用户连接的 6 步

How it works
📌 本节核心要点

当用户把 MCP 客户端连接到门户时，发生 6 步——(1) MCP 客户端向门户 URL（https://<subdomain>.<domain>/mcp）发请求；(2) Cloudflare Access 通过浏览器 OAuth 2.0 流程（或服务令牌）认证用户；(3) 门户建立 MCP 会话，返回所有已启用上游服务器的工具列表；(4) 用户调用工具时，门户根据工具命名空间识别目标上游服务器，附加相应凭据（用户 OAuth 令牌或管理员凭据），代理请求；(5) 如果启用网关路由，出站请求在到达上游服务器之前经过 Cloudflare Gateway 做 HTTP 日志记录和 DLP 检查；(6) 上游响应返回 MCP 客户端。

这是产品的核心数据流。文档写得很清晰——

🔄 完整连接流程
  • 步骤 1——MCP 客户端向门户 URL 发请求——格式 https://<subdomain>.<domain>/mcp
  • 步骤 2——Cloudflare Access 通过基于浏览器的 OAuth 2.0 流程（或服务令牌标头）对用户身份验证
  • 步骤 3——门户建立 MCP 会话，返回所有已启用上游服务器的可用工具列表。注意——只有用户能过 Allow Access 策略的服务器才会在工具列表里出现
  • 步骤 4——用户调用工具时，门户根据工具命名空间识别目标上游 MCP 服务器，附加相应凭据（用户 OAuth 令牌或管理员凭据），代理请求
  • 步骤 5——如启用网关路由，出站请求在到达上游服务器前经过 Cloudflare Gateway 进行 HTTP 日志记录和 DLP 检查
  • 步骤 6——上游服务器响应返回 MCP 客户端

关键设计——门户做"凭据注入"——上游服务器只看到一个统一的"管理员凭据"或"用户 OAuth 令牌"，不需要知道是哪个 AI agent 在调。这给 Cloudflare 留出做集中策略和审计的空间。

会话生命周期——每个 MCP 客户端连接创建一个会话，持续到用户断开或不活动 24 小时过期。会话期间，用户无需断开连接即可启用或禁用单个服务器。

文档还提到一个有趣的命名沿革——"MCP 服务器门户之前被称为代理网关"——API 路径、Terraform 资源和内部代码库可能仍用 agents_gateway 或 agw 前缀。产品名称改为 MCP 服务器门户，仪表板导航在AI 控件下。这暗示产品正在"agent 网关 → MCP 网关"演化中。

CHAPTER 04 · TRANSPORT

传输协议 · 自动检测 4 种连接

Transport detection
📌 本节核心要点

MCP 协议早期有 SSE（Server-Sent Events）和 Streamable HTTP 两种传输。Cloudflare 门户用 Streamable HTTP 或 SSE 连接上游 MCP 服务器，无需指定——按 URL 模式自动检测。URL 以 /mcp 结尾——仅尝试 Streamable HTTP；以 /sse 结尾——优先 SSE（启用网关路由时改为 Streamable HTTP）；其他——按"原始 URL Streamable HTTP → 原始 URL SSE → {url}/mcp Streamable HTTP → {url}/sse SSE"顺序尝试。返回 404、405、406 错误时回退到下一策略，其他错误终止。这是处理协议演化期的优雅工程妥协。

Streamable HTTP SSE Transport 404/405/406 Fallback

MCP 协议在 2024 年早期用 SSE（Server-Sent Events）作为主传输——后来升级到 Streamable HTTP 作为更现代的标准。但过渡期是混乱的——很多 MCP 服务器仍在 SSE，一些已经在 Streamable HTTP，一些两个都支持。

Cloudflare 的解法非常实用——不要求用户指定上游服务器使用哪种传输，按顺序自动检测——

上游 URL 模式连接策略（按顺序）
以 /mcp 结尾仅 Streamable HTTP
以 /sse 结尾SSE（启用网关路由时改为 Streamable HTTP）
其他 URL原始 URL Streamable HTTP → 原始 URL SSE → {url}/mcp Streamable HTTP → {url}/sse SSE

如果连接尝试返回 404、405 或 406 错误——门户回退到下一策略。其他错误都会导致连接终止。

这是处理协议演化期的优雅工程妥协——"你别管什么传输，我帮你试"。这种"友好默认 + 可解释失败"是好的企业产品该有的样子。

CHAPTER 05 · BUILT-IN TOOLS & CODE MODE

内置门户工具 + Code Mode

Built-in tools & Code Mode
📌 本节核心要点

每个门户向 MCP 客户端公开 3 个内置工具——portal_list_servers 列出所有上游服务器；portal_toggle_servers 打开 URL 服务器选择页；portal_toggle_single_server 按 ID 启用/禁用单服务器。启用上下文优化后增加 portal_query_tools（搜索工具返回定义）和 portal_execute（搜索并执行）。但最重要的是 Code Mode——默认启用——把所有上游工具合并为一个 code 工具。AI 代理写 JavaScript 调用每个工具的类型化方法，代码在隔离的动态 Workspace运行。"无论有多少工具可用，上下文窗口使用情况都能保持固定"。这是对 Anthropic 'code execution as tool' 哲学的认同。

每个门户向 MCP 客户端公开 3 个内置工具——

· portal_list_servers——列出所有可用上游服务器及其 ID、名称以及当前是否开启
· portal_toggle_servers——打开 URL 服务器选择页面
· portal_toggle_single_server——按服务器 ID 启用/禁用单个服务器，无需离开 MCP 客户端

启用上下文优化后，根据模式不同会增加额外工具——详见第 7 章。

但最重要的——Code Mode——

所有门户默认启用 Code 模式。

它将所有上游工具合并为一个 code 工具。

AI 代理编写 JavaScript 代码，
调用每个工具的类型化方法，
代码在隔离的动态 Workspace 环境中运行。

这样——无论有多少工具可用——
上下文窗口使用情况都能保持固定。

— CODE MODE

这是非常 Anthropic 风格的设计——"用代码执行作为单一工具"。它的好处——

· 上下文恒定——不管你有 10 个 MCP 还是 100 个 MCP，客户端看到的工具列表永远只有 1 个 code
· 组合性——AI 能用 JavaScript 把多个工具调用编排在一段代码里，比一次次 round-trip 高效
· 类型安全——Workspace 提供类型化方法签名，编译时就能发现错误，不用浪费 LLM 的"试错调用"

"Code Mode"的本质——把"工具调用"提升到"代码执行"——更像"程序员的 REPL"而不是"每次只能按一个按钮的遥控器"。这与 Anthropic Claude API 最近推的 "programmatic tool calling" 在哲学上是完全一致的。

CHAPTER 06 · ALIASES

工具别名 + 别名优先级

Tool aliases
📌 本节核心要点

别名解决三个问题——(1) 替换含义不明确的工具名；(2) 加 AI 代理能更好选择正确工具的描述；(3) 跨多 MCP 服务器实现命名标准化。别名长度 1-40 字符，只含字母数字、连字符、下划线，必须以字母数字开头结尾，匹配 ^[a-zA-Z0-9]+([_-][a-zA-Z0-9]+)*$。同服务器上两个工具不能同名。别名优先级——portal_alias（门户级）> server_alias（服务器级）> alias> 原始工具名。描述同样遵循优先级——portal_description 覆盖 server_description。这种分层覆盖设计给企业管理员极强的灵活性。

别名功能看起来微不足道但极其有用——文档列出 3 个用途——

· 将含义不明确的工具名替换为符合贵组织术语的名字
· 添加/改进描述，让 AI 代理能选择正确工具（这是第 1 个 Thariq 文章 "Claude 似乎喜欢调这个工具" 原则的工业级落地）
· 跨多 MCP 服务器实现命名标准化

规则——别名长度 1-40 字符，只能含字母、数字、连字符、下划线。必须以字母数字开头和结尾。匹配正则 ^[a-zA-Z0-9]+([_-][a-zA-Z0-9]+)*$。同一服务器上任何两个工具或提示不能用相同名称，不管是别名还是原始上游名。

例如：search_customer_records、get-user-profile——都是合法别名。

📐 别名优先级

可以在两个级别设置别名——

  • 服务器级——alias 字段——适用于包含此服务器的所有门户
  • 门户级——portal_alias 字段——仅在特定门户内有效；覆盖服务器级

当存在多个名称时——门户按以下顺序解析——portal_alias > server_alias > alias > 原始工具名。

描述同样遵循此优先级——可在 updated_tools 条目中包含 description 字段。API 响应中——服务器级描述以 server_description 返回，门户级描述以 portal_description 返回。同时设置时，门户级覆盖服务器级。

这种分层覆盖设计给企业管理员极强的灵活性——"在不改上游 MCP 服务器的前提下，给我企业内部统一的命名和描述"。

CHAPTER 07 · CONTEXT OPTIMIZATION

上下文优化 · 4 种模式

Context optimization modes
📌 本节核心要点

这是这份文档对 AI 工程师最有价值的一节——Cloudflare 把"上下文窗口稀缺"作为产品功能正式 expose。4 种模式从最浪费到最节省——(1) Full tools（默认）——所有工具定义全部加载；(2) Minimize tools——只显示工具名+短描述，模型可用 portal_query_tools 按需取完整定义；(3) Search and execute——只暴露 portal_query_tools + portal_execute，所有工具都"按需发现 + 通过代理执行"；(4) Code Mode——所有上游工具合并为单一 code 工具（第 5 章）。这一组模式就是 Anthropic Thariq 文章 "Progressive Disclosure" 原则的产品化落地。

这是这份文档对 AI 工程师最有价值的一节。Cloudflare 把"上下文窗口稀缺"作为产品功能正式 expose——4 种模式从最浪费到最节省——

⚡ 4 种上下文优化模式
  • (1) Full tools（默认）——所有工具的完整定义全部加载到客户端 context。简单但贵——50 个 MCP 服务器、每个 20 个工具 = 1000 个工具定义全塞进 context window。
  • (2) Minimize tools——只暴露工具名 + 短描述。模型需要详细信息时调 portal_query_tools——通过正则表达式模式搜索工具并返回完整定义。"按需取详情"。
  • (3) Search and execute——更激进——只暴露两个工具——portal_query_tools（搜索）和 portal_execute（执行）。所有上游工具都通过这两个间接访问——模型搜索找到合适工具，然后调 portal_execute 执行。
  • (4) Code Mode（默认启用）——最激进——所有上游工具合并为单一 code 工具——AI 写 JS 调用类型化方法（详见第 5 章）。

这4 种模式构成一个完整的"工具暴露策略谱系"——从"全摊开，简单直接"到"全藏起来，按需发现"。企业可以根据 agent 任务复杂度和 context 预算自由选择。

这一组功能是Anthropic 在 Thariq 文章里推的 "Progressive Disclosure" 原则的产品化落地——Cloudflare 把它做成了企业级 MCP 门户的核心卖点。

CHAPTER 08 · EDITOR'S NOTE

广哥的总结 · MCP 进入企业级

MCP goes enterprise

读完这份产品文档，我有几个感受——

第一——MCP 协议正式进入"企业基础设施"阶段。MCP 2024 年由 Anthropic 发明，2025 年捐给 Linux Foundation，2026 年 6 月 Cloudflare 推出企业级网关。这跟 HTTP 的演化路径一模一样——协议先在开发者社区扩散，然后大厂下场做"反向代理 + 集中策略"层。

第二——Cloudflare 这次没有发明新协议——而是把自家最有价值的能力（Access、Zero Trust、Gateway、DLP）"白嫖"接进 MCP 生态。这是非常聪明的战略——不跟 Anthropic 抢标准制定权，而是抢"标准上的网关层"。这跟 Cloudflare 当年对 HTTP 做的事是同样套路——它不发明 HTTP，但你用 HTTP 时很难绕开它。

第三——"Code Mode"是这个产品最有思想性的设计。把所有 MCP 工具抽象成单一 code 工具 + 类型化方法——本质上是"把 MCP 协议升级成程序化接口"。这跟 Anthropic 自己最近推的 "programmatic tool calling" 在哲学上完全一致。我赌——这会是未来 MCP 客户端的默认模式。

第四——"上下文优化"被产品化是个里程碑。三个月前，我们还把 context window 当作"模型能力的副产品"。现在 Cloudflare 把它当作"企业管理员可选的 4 种模式"。context engineering 正在从"模型能力问题"变成"网关产品能力"。

第五——MCP 门户文档里那句"AI 模型接触到的外部上下文越少，其响应往往就越好"——这就是 Thariq 在 X 帖子里的洞察、MBZUAI 论文里的"Progressive Disclosure"——同一个工程哲学。整个 AI agent 工程界在 2026 年达成了一个共识——"context 是最稀缺资源；少即是多"。

MCP 2024 年由 Anthropic 发明——
2025 年捐给 Linux Foundation——
2026 年 6 月 Cloudflare 推出企业级网关。

这跟 HTTP 的演化路径一模一样——
协议先在开发者社区扩散，
然后大厂下场做"反向代理 + 集中策略"层。

下一步——MCP 应用商店、MCP CDN、MCP DDoS 防护⋯⋯
你能想到的所有"HTTP 时代有过的层"，
MCP 时代都会重新发生一遍。
